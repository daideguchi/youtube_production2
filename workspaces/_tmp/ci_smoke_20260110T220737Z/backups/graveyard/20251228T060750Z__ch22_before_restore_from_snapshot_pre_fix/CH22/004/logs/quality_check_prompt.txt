## 1) 最重要ルール（音声品質の下限）

### 1.1 ポーズ挿入の唯一の記号
- **ポーズ挿入として許可される記号は `---` のみ**。
- `---` は **1行単独**で置く（前後は改行）。意味の切れ目（話題転換/場面転換/呼吸の区切り）にだけ使う。
- 原則: **セクションが切り替わる境界**（設計図/構成上の区切り）で `---` を置く（TTS側の0.5秒ポーズに対応）。
- 音声側の解釈（strict）:
  - 通常のつなぎは **0.1秒**（デフォルト）
  - `---` は **0.5秒**の無音ポーズ
  - 空行/改行はポーズ指示として扱わない（整形のために入れてもよい）
- `***` や `___`、`///`、`===` 等の区切り記号は使用しない（音声の不自然な途切れや事故の原因）。

### 1.2 記号・括弧の過剰使用を禁止（TTSが不自然に切れやすい）
- `「」` / `『』` と `（）` は **多用しない**（必要な場面だけ）。原則として **直接話法より間接話法** を優先する。
- 引用はできるだけ `「」` / `『』` を使わず、地の文で言い換える（例: “ある人はこう言いました” の形）。
- 目安（TTS事故防止）: 1本あたり `「」` / `『』` は **合計20個（10組）以内**、`（）` は **合計10個（5組）以内**。超える場合は意味を保ったまま言い換える。
- 運用上の扱い: この「目安」は **script_validation のハード上限**として扱う（超過は不合格）。
  - 上限は `configs/sources.yaml` の `script_globals.a_text_quote_marks_max` / `script_globals.a_text_paren_marks_max` が正本（必要ならチャンネル別override可）。
  - 超過した場合は **自動クリーナ（非LLM）で括弧だけ外して基準内に落とす**（意味は維持・全文リライト禁止）。
  - その結果、字数が下限を割った場合は **Extend/Expand（追記のみ）で回復**する（リトライ回数で押し切らない）。
- 会話の連打でテンポを作らない。台詞が必要なら **短く**、地の文に溶かし込む。
- ルビ/読み仮名/注釈目的の括弧は原則禁止（読み上げが破綻しやすい）。

### 1.3 視聴者ストレスを生む文章を禁止
- 淡々とした繰り返し構文、同じ言い回しの連発、蛇足の比喩、回りくどい前置き、過剰な煽りを避ける。
- 断定・説教・脅し・人格否定はしない。必要なら事実と提案に落とす。
- “言い換えで水増し” をしない。厚みは **具体の追加**で作る（状況・行動・心の動き・結果）。
- **同一段落の丸ごと重複は禁止**（コピー/生成ループ由来の反復）。  
  目安（機械チェック）: 120文字以上の段落が同一（空白差を無視して一致）なら不合格。

### 1.4 捏造・虚構の禁止（信頼を壊さない）
- 根拠のない統計・割合（`％` / `パーセント`）・研究結果・大学名/研究所名・論文/実験の作り話を本文に書かない。
- 事実確認ができない固有名詞や数値は使わず、必要なら一般化して述べる（例: “研究では〜” の断定や数字は避ける）。
- 科学/医学の橋渡しは、**安全な一般論を短く1段落**に留める（呼吸と緊張、書くことと客観視など）。数字・機関名・論文名は出さない。

### 1.5 「字数合格だけ」を禁止（LLM品質ゲート必須）
- 文字数や禁則がOKでも、内容が薄い/冗長/流れが不自然なら **不合格**とする。
- フロー/自然さの合否は **機械的ルールではなく推論モデルの判定（LLM Judge）**で担保する。
- 運用の正本: `ssot/ops/OPS_A_TEXT_LLM_QUALITY_GATE.md`（Judge→Fixer の2段階で最終品質を固定）。

### 1.6 終わり方（未完を禁止）
- Aテキストは **最後まで完結**していること（途中で切れて終わるのは不合格）。
- 目安（機械チェック）:
  - 最後の有効文字が `。` / `！` / `？` / `!` / `?` / `」` / `』` などで閉じている。
  - 助詞や語尾の途中（例: `〜になり` / `〜で` / `〜が`）で止まらない。
- これは「決まり文句を入れろ」ではなく、**ぶつ切り事故（生成途中で止まる）を確実に検知して止める**ためのルール。

### 1.7 文字の健全性（外字/制御文字を禁止）
- `�`（U+FFFD, Unicode replacement character）が混入している場合は不合格（コピー事故/文字化けの典型）。
- 目に見えない制御/整形文字（例: ゼロ幅スペース等）や、私用領域（PUA）等の特殊文字が混入している場合は不合格。
- 明らかな誤字・外字が検出された場合は不合格（TTSの読み崩れ/視聴者ストレスの原因）。  
  既知の混入パターンは `script_validation` の決定論クリーナで **置換修復**する（全文リライトは禁止）。

## 2) Aテキストの禁止事項（混入事故を根絶）

台本本文（Aテキスト）に入れない:
- URL（`http(s)://` / `www.`）
- 出典メタ、脚注、参照リンク、引用番号（例: `[13]` / `([xxx][13])`）
- 箇条書き/番号/記号リスト（`-` / `・` / `1.` / `A.` 等）
- アウトライン由来のメタ（例: `約600字`、`A. 見出し - 約600字` などの「字数メモ」）
- Markdown装飾（例: `**太字**`）
- 設定/CSVデータ/構成案/プロット等の **制作メモの貼り付け**（例: `設定` 見出し＋ `主人公: ...` の羅列）
- エピソード見出し行（例: `CH23-001: ...`）
- タイムスタンプ、チャプター表、制作裏（AI/収録/台本などのメタ言及）
- 章立てメタ（例: `第3章` / `第3章を始めましょう` のような“章見出し”）

※出典は本文ではなく `content/analysis/research/references.json` 等の research 側へ集約する。

---

## 3) 読みやすさ（耳で理解できる最小設計）

- 1段落は1話題。2〜4文を目安に詰め込みすぎない。
- 1文は長くしすぎない（息継ぎできる長さ）。主語が迷子にならないように整える。
- 専門語は最小限。出すなら直後にやさしい言い換えを添える（定義の長話はしない）。
- 比喩は “一度で伝わる短さ” に限定し、連打しない。

## 4) チャンネル別ルールとの優先順位（衝突時の扱い）

1. 本書（全チャンネル共通のAテキスト品質）  
2. チャンネル固有の `script_prompt.txt` / Persona / 企画CSV  
3. 作業メモ・一時指示

補足:
- チャンネル `script_prompt.txt` に構成/形式/記号の指示が混ざっていても、Aテキスト向けLLM呼び出しでは衝突しやすい要素を落とした `a_text_channel_prompt`（派生）を用いる。骨格は `ssot/ops/OPS_SCRIPT_PATTERNS.yaml` が正本。

例外が必要な場合:
- チャンネル固有の理由と例外条件を **SSOTに明記**してから適用する（無断で例外を増やさない）。


【対象コンテンツ】
昔は玄関先や縁側で、顔を合わせてゆっくり話していました。隣の家の方が洗濯物を干していると、自然に声をかけ合う。向かいの奥さんが買い物から帰ってくると、ちょっと立ち止まって天気の話をする。そんな何気ないやりとりが、毎日の暮らしに溶け込んでいたのです。

会話には間がありました。言葉と言葉の間に、相手の表情を見る時間がたっぷりとあった。うなずきや笑み、あるいはちょっと困ったような顔。それらすべてが、今どのくらい話せばよいのか、どのくらいの距離がちょうどよいのかを、そっと教えてくれていたように思います。お互いの都合も、顔を見ればだいたいわかった。忙しそうなら早めに切り上げるし、余裕がありそうならもう一服お茶をすすめる。そんな調整が、無言のうちにできていたのです。

それが今では、LINEのグループが、かつての井戸端会議の場を肩代わりしています。確かに便利です。遠くに住む友人ともすぐに連絡が取れる。町内会の連絡も、わざわざ回覧板を回さなくて済む。時間や場所を選ばずにやりとりできるのは、本当にありがたいことです。

でも、少しずつ感じる違和感がありました。送ったメッセージに付く「既読」の二文字に、私たちの心は一喜一憂してしまうようになったのです。あの小さな文字が、読まれた証であり、同時に私たちの関係の温度を測るものさしのように感じられてきました。

既読がすぐにつくと、ほっとする。なぜか遅いと、もしかして嫌われたのかな、と不安になる。たった二文字の表示が、一日の気分を左右することさえある。そう気づいたとき、ふと昔を思い出しました。あの頃は、既読も未読もありませんでした。代わりにあったのは、相手の目の動きや、声の調子、そして何よりその場の空気感そのものだったのです。

技術は確かにコミュニケーションの形を変えました。顔を合わせなくても話せるようになった。その便利さは、否定しようのない現実です。しかし同時に、距離感の測り方に新たな「記号」が生まれました。それが既読表示です。読んだか読んでいないか、という事実だけが、まるで関係のすべてであるかのように、時に大きく映し出されてしまう。

私たちはその記号に、どれだけの意味を込めてしまっているのでしょう。あの人は忙しいだけなのに、私のことが嫌いなんだと思い込んでしまう。グループの中で自分だけ既読が遅いと、仲間外れにされているような気がして胸が苦しくなる。かつては、相手の表情や様子から自然に汲み取っていたことが、今ではたった一つの記号に委ねられてしまったかのようです。

もちろん、LINEのグループは悪いものではありません。離れていても繋がっていられる、それは素晴らしいことです。問題は、技術そのものではなく、私たちのその受け止め方にあるのでしょう。便利な道具が、いつの間にか私たちの心の動きまで規定し始めている。その変化に、そっと気づく時が来ているのかもしれません。

昔の井戸端会議には、強制も義務もありませんでした。都合が合えば話し、合わなければまた今度。それが自然な距離感でした。今、私たちは画面上の記号に、あの頃の自然さを無意識に求めているのではないでしょうか。しかし、機械が示す既読と、人間の心の間には、どうしても埋められない隙間がある。その隙間に、不安や寂しさが入り込む余地が生まれてしまったのです。

変化そのものを静かに観察してみましょう。技術が進み、生活が変わる。それは悪いことばかりではありません。ただ、それに合わせて私たちの心のあり方も、少しずつ調節していく必要がある。そう思うのです。かつては顔を見て感じ取っていたものを、今では画面上の記号で推し量ろうとしている。そのギャップに、そっと目を向けてみる。それだけで、少し楽になることがあります。

便利さと引き換えに、何かを見失ってしまったわけではないのです。ただ、新しい形のコミュニケーションに、私たちがまだ慣れていないだけ。かつての井戸端会議と今の既読表示、その両方の良さを、静かに見つめ直す時なのかもしれません。

では、その「既読」が付かないとき、私たちは何を考えてしまうのでしょうか。

---

スマホの画面に、送ったメッセージがただ一つ、既読になっていない。他の方たちへの連絡には、さっさと既読マークが並んでいる。この違いだけが、静かな部屋の中ではっきりと浮かび上がる。胸のあたりに、ちくりとした違和感が広がります。もしかして、私だけが避けられているのか。何か気に障ることを言ってしまったのだろうか。そんな思いが、頭の中をぐるぐると回り始めます。

---

しかし、少しだけ手を止めてみましょう。その既読が付かないという現象は、本当に人間関係の悪化だけを示しているのでしょうか。実は、もっと別の理由が隠れていることが少なくありません。スマホという機械を通したコミュニケーションには、私たちが思う以上に、単純な技術的な落とし穴がいくつもあるのです。

例えば、通知の設定が変わってしまった場合。自分では気づかないうちに、特定のトークルームの通知がオフになっているかもしれません。あるいは、アプリ自体に一時的な不具合が起きている可能性もあります。最近では、スマホの機種変更やOSのアップデートがきっかけで、こうした小さな齟齬が生じることも珍しくありません。それに、トークルームを誤って退出してしまい、気づかずに一人だけ外れている状態というケースも、実際に耳にします。

では、どうすればいいのか。まずは落ち着いて、一つだけ確認できることがあります。それは、他の方々へのメッセージでも、同じように既読が付かない状態なのかどうか、ということです。グループ内で別の方に個別にメッセージを送ってみる。あるいは、ごく簡単なやりとりを試してみる。それだけで、状況が見えてくることがあります。

もし、他の方へのメッセージにはすぐに既読が付くのであれば、それはあなただけに向けた変化かもしれません。でも、どのメッセージにも反応が遅い、あるいは全体的に既読が付きにくいのであれば、それはアプリや設定、あるいは通信環境といった技術的な側面に原因がある可能性が高まります。この一手は、決して大げさな行動ではありません。ほんの少しの確認で、必要以上に自分を責めたり、不安に駆られたりする気持ちに、静かなブレーキをかけることができます。

私たちはつい、自分の送ったメッセージにだけ既読が付かないと、自分の人間関係に問題があると決めつけてしまいがちです。でも、画面の向こうには、機械の仕組みや設定の複雑さが横たわっています。それを一度、冷静に見つめ直してみる。それが、感情の渦に巻き込まれる前に、自分自身を守る最初の一歩になるのです。

---

技術的な確認を経て、もしそれが単なる不具合だったと分かれば、胸のつかえは少し軽くなるでしょう。でも、やはり気になるのは、その先にある関係性の小さなサインかもしれません。既読という記号の向こう側には、もっと微かなコミュニケーションの跡が残されているからです。

---

では、もう一度画面に目を落としてみてください。
メッセージを送ったあと、私たちの目はつい、その横に小さく表示される「既読」の二文字を探してしまいます。
読まれた、読まれていない。
たったそれだけのことが、なぜか気になって仕方がない。
でも、ちょっと待ってください。
その「既読」という表示の、ほんの数センチ下、あるいはその会話の少し前や後を、ゆっくりと見渡してみませんか。
そこには、もっとたくさんの、静かなサインが息づいているはずです。

既読表示は、あくまで事実の報告でしかありません。
メッセージが相手のスマホの画面に表示された、という機械的な確認です。
そこには、読んだ人の気持ちや、その後の関係性は、一切映り込んではいない。
私たちが本当に知りたいのは、そこから先のことではないでしょうか。

たとえば、スタンプです。
あなたが送ったメッセージに、誰かがスタンプ一つ、ぽんと押してくれている。
それは、もう既読をはるかに超える、小さな返事です。
「了解しました」「そうですね」「ありがとう」。
たった一つの絵文字が、それらの言葉を代弁してくれています。
逆に、スタンプもなく、そのまま会話が流れていくこともあるでしょう。
それは必ずしも無視や拒絶ではなく、ただ、その話題に追加する言葉が見つからなかっただけかもしれない。
スタンプの有無は、単純な記号以上の、会話の温度を伝えるサインの一つです。

もう一つ、よくある光景です。
あなたへの直接の返事はなくても、そのすぐ後、あるいは別の日に、同じグループ内で会話がぽつりぽつりと続いている。
今日の天気の話、近所で見かけた花の話、孫のちょっとした出来事の話。
その流れに、あなたも自然に一言、加えることができる。
そこには、既読か未読かという枠組みでは測れない、緩やかな時間の共有があります。
「私だけ既読がつかない」と感じたそのメッセージは、実は大きな川の、ほんの一滴の水だった。
川そのものの流れは、少し下流でも、しっかりと続いているのです。

私たちは、一本の木の、特定の枝ばかりを見つめていることがあります。
その枝にだけ、葉っぱが付いていない、実がなっていない。
それだけで、この木全体が枯れかかっているのだと、早合点してしまう。
でも、視線を少し上げて、木全体を見回してみれば、他の枝には青々とした葉が茂り、別の場所には小さな蕾さえついているかもしれない。
コミュニケーションという森の中では、既読表示は、一本の枝に過ぎません。
スタンプの返信、別の話題での会話の続き、グループ内の緩やかな空気感。
それらすべてが、その関係という木の、現在の姿を形作っています。

一つの記号にすべての意味を託そうとすると、心は簡単に振り回されてしまいます。
既読が付かないのは自分のせいだ、嫌われたのだ、仲間外れにされたのだ。
そうした思いは、たった一つのデータから、巨大な物語を紡ぎ出してしまう力があります。
けれども、もしその物語が本当だとすれば、それは既読以外のあらゆる場所にも、同じような冷たさが表れているはずではありませんか。
他の話題でも完全に無視される、スタンプも一切返ってこない、まるでそこにいないかのように扱われる。
そうしたことが重なって初めて、それは一貫したサインと言えるでしょう。
