【執筆ルール（絶対）】
- 最優先は「YouTubeの読み台本として自然で、内容が頭に入る」こと。情緒で引き伸ばさず、理解が増える具体で厚みを作る。
- 次の言い回しで始まる段落は作らない（GPT臭になりやすい）:
  - 朝の光/朝の静けさ/湯気/波紋/港/灯り/霧/冷たい風/矢/塩を塗る などの情景・比喩で“それっぽく”始める導入
- 導入は必ず「視聴者が今困っている場面」から入る。相手に正しいことを言われて胸が痛くなる、言い返したくなる、沈黙して後悔する、などの生活の場面を具体に置く。
- 設計図（PLAN_JSON）が最優先。企画メモ/ペルソナ/チャンネル固有プロンプトに矛盾があっても、設計図の `core_message` / `sections` / `core_episode` を優先し、話題を増やして帳尻合わせしない。
- `---` はポーズ挿入としてのみ使用し、1行単独で置く。他の区切り記号は禁止。
- URL/脚注/参照番号/箇条書き/番号リスト/見出しは禁止。
- `『』` は出力に含めない（強調に使わない。引用にも使わない）。
- `「」` と `（）` は極力使わない（必要最小限）。可能なら必ず地の文で言い換える。
  - 例: `「どうでもいい」` ではなく `どうでもいいという言葉` / `どうでもいいと心の中で言う` のように書く。
- 作り話感の強い人物例（年齢/職業/台詞の作り込み、最後を「〜そうです」で締める等）は禁止。必要なら一般化して自然に書く。
- 「報告されています」「ワークショップ」などの“それっぽい権威づけ”で説得力を作らない（事実リスク）。
- 抽象や雰囲気の水増し、詩的な比喩、同趣旨の言い換え連打は禁止。
  - NG例: 情景や比喩だけが続き、何をどうするかが増えない段落。こういう段落は書かない。
- 具体例は設計図の `modern_examples_policy.max_examples` を厳守する（例の連打は禁止）。
- 一文を長くしすぎない。耳で理解できる長さで切る。
- 文字数は必ず min〜max に収める（改行は文字数に数えない）。
- 設計図に `core_episode.safe_retelling` がある場合:
  - 中心エピソードはそれ1つを中心にする（別の逸話・別の教えに話題を分散させない）。
  - `safe_retelling` を地の文で言い換え、その筋を崩さずに深掘りする。`must_include` の要点は必ず入れる。
  - 本文に無い専門用語や教義名を新規に増やさない。必要なら日常語に言い換える（概念語の追加で“広げない”）。
  - 出典/作品名/年代/地名/固有名を断定しない。史実風の細部を足して“それっぽく”作り込まない。

【構成ルール】
- 設計図の sections の順で書く。
- sections の境目に `---` を1行単独で入れる。
- `---` の合計本数は <<PAUSE_MARKERS_REQUIRED>> 個にする（不足/過剰を避ける）。
- 各セクションは設計図の char_budget を「目安」ではなく「達成目標」として扱う。
  - 目安: 各セクションは char_budget の 80〜125% に収める（極端に短いセクションを作らない）。
- もし短くなりそうなら、結論の言い換えで伸ばさず、理解が増える具体（状況→心の動き→核心の見立て/気づき→行動）を追加して厚みを作る。
- 終盤で急に別の話題や例を増やさず、結びに向けて自然に収束させる。
